# Chapter 10 — Complete Research Brief application

## Objective

Run the complete offline system and connect every earlier concept without exposing all detail at once.

## Interactive lab

Submit a research question. The application returns the outcome first. Expand plan, evidence, public events, usage and evaluation only when needed.

The integrated service:

1. Validates and plans the question.
2. Runs the bounded agent with the simulated provider.
3. Executes only the registered local-corpus tool.
4. Accepts only known evidence citations.
5. Stops honestly when evidence is insufficient.
6. Estimates abstract usage and latency.
7. Includes deterministic evaluation evidence.

## Exercise

Run one supported and one unsupported question. Compare plan status, evidence, limitations and terminal state. Then set the tool budget to zero in code and predict the result.

## Common mistakes

- Replacing the teaching progression with one complex dashboard.
- Claiming integration means production readiness.
- Hiding limitations after a successful run.
- Maintaining separate demo logic that disagrees with the tested engine.

## Knowledge check

**What would you change for production?**

Add authenticated multi-user isolation, durable transactional stores, real-provider adapters with usage reporting, expiry and identity for approvals, distributed tracing, stronger content-security controls, deployment hardening and operational monitoring. The educational core deliberately does not claim these features.
